# 代码审查 Skill (Code Review)

当用户请求代码审查时，按照以下流程和检查清单对指定的文件或模块进行全面审查。

## 审查流程

1. **确定范围**：确认要审查的文件或模块。若用户未指定，则审查最近修改的文件。
2. **逐项检查**：按照下方检查清单逐一核查。
3. **生成报告**：将发现的问题按严重程度分类，输出结构化的审查报告（作为 Artifact）。
4. **提出修复建议**：对每个问题给出具体的修改方案或代码示例。

## 检查清单

### 1. TypeScript 类型安全

- [ ] 不使用 `any` 类型（除非有充分理由并加注释说明）
- [ ] 函数参数和返回值有明确的类型注解
- [ ] 接口/类型定义置于 `shared/types/` 或就近的 `types.ts` 中
- [ ] 使用联合类型而非枚举字符串魔法值（例：`type Status = 'idle' | 'loading' | 'error'`）
- [ ] 泛型使用得当，避免过度泛化
- [ ] `as` 类型断言尽量少用，优先使用类型守卫（type guards）

### 2. React 组件规范

- [ ] 组件使用函数式组件 + Hooks
- [ ] Props 通过独立的 `interface` 或 `type` 定义
- [ ] 避免在 JSX 中定义内联函数（用 `useCallback` 包裹）
- [ ] `useEffect` 依赖数组完整且正确
- [ ] 避免不必要的 `useState`，派生状态优先用 `useMemo`
- [ ] 组件职责单一，超过 200 行应考虑拆分
- [ ] 事件处理函数以 `handle` 开头（如 `handleClick`、`handleSubmit`）
- [ ] 条件渲染逻辑清晰，避免深层嵌套三元表达式

### 3. 状态管理

- [ ] 全局状态使用项目约定的 store 模式（检查 `src/features/` 下的 store）
- [ ] 避免 props drilling 超过 2 层
- [ ] 异步状态（loading / error / data）处理完整
- [ ] 状态更新不可变（immutable），不直接修改对象或数组

### 4. 后端 API 与安全

- [ ] API 路由使用 RESTful 命名约定
- [ ] 请求参数有验证和清洗（防止注入）
- [ ] 错误处理使用 `try-catch`，返回统一的错误响应格式
- [ ] 敏感信息（API key 等）只从环境变量读取，不硬编码
- [ ] 异步操作正确使用 `async/await`，无未捕获的 Promise
- [ ] 中间件逻辑与业务逻辑分离

### 5. AI Agent / Prompt 质量

- [ ] Prompt 模板没有冗余信息，控制 token 消耗
- [ ] 使用模板字面量清晰组织 prompt 结构
- [ ] AI 响应有 JSON 解析和错误处理（`try-catch`）
- [ ] 有降级/重试逻辑处理 API 调用失败
- [ ] `@图像` 标签引用格式正确（`[@图像_名称#id]`）
- [ ] Agent 间的数据流清晰，无循环依赖

### 6. 性能

- [ ] 列表渲染使用 `key` 属性且值稳定
- [ ] 大列表考虑虚拟化（项目已使用 `react-virtuoso`）
- [ ] 避免在循环中进行 DOM 操作或 API 请求
- [ ] 图片和资源有懒加载处理
- [ ] 避免不必要的重渲染（检查 `React.memo`、依赖数组等）
- [ ] 后端避免同步 I/O 阻塞事件循环

### 7. 错误处理与健壮性

- [ ] 所有 `fetch` / API 调用有 `catch` 处理
- [ ] 空值检查（`null`、`undefined`）到位
- [ ] Optional chaining (`?.`) 和 nullish coalescing (`??`) 使用得当
- [ ] 用户输入有边界校验
- [ ] 文件导入/导出（ZIP 功能）有异常处理

### 8. 代码风格与可维护性

- [ ] 函数和变量命名语义化，见名知意
- [ ] 无残留的 `console.log`（调试完成后应清理）
- [ ] 无注释掉的代码块（应删除，Git 有历史记录）
- [ ] 复杂逻辑有必要的注释说明
- [ ] 文件/模块结构遵循项目约定的分层架构
- [ ] 单个函数不超过 50 行，超出应拆分
- [ ] 无重复代码（DRY 原则），相似逻辑抽取为工具函数

### 9. 测试覆盖

- [ ] 核心业务逻辑有对应的 Vitest 单元测试
- [ ] 关键用户流程有 Playwright E2E 测试覆盖
- [ ] 测试用例覆盖正常路径和异常路径
- [ ] Mock 数据贴近真实场景
- [ ] 测试文件与源文件位置对应（`tests/` 目录）

## 审查报告格式

审查完成后，生成一份 Markdown 报告，格式如下：

```markdown
# 代码审查报告

## 审查范围
- 文件列表...

## 发现汇总

| 严重程度 | 数量 |
|---------|-----|
| 🔴 严重  | N   |
| 🟡 警告  | N   |
| 🔵 建议  | N   |

## 详细发现

### 🔴 严重问题
1. **[文件名:行号]** 问题描述
   - 影响：...
   - 建议修复：...

### 🟡 警告
1. **[文件名:行号]** 问题描述
   - 建议修复：...

### 🔵 改进建议
1. **[文件名:行号]** 问题描述
   - 建议修复：...

## 亮点（可选）
- 代码中做得好的地方...
```

## 严重程度定义

| 级别    | 说明                             | 示例                                       |
| ------- | -------------------------------- | ------------------------------------------ |
| 🔴 严重 | 可能导致崩溃、数据丢失、安全漏洞 | 未捕获的异常、API key 泄露、SQL 注入       |
| 🟡 警告 | 可能导致 bug 或性能问题          | 缺失的空值检查、内存泄漏、不完整的依赖数组 |
| 🔵 建议 | 提高代码质量和可维护性           | 命名不规范、缺少类型注解、可以提取工具函数 |
